# Защита FastPanel и сервера

## Для чего нужен раздел

Этот раздел объясняет, как безопасно использовать FastPanel или другую панель управления сервером для обменника. Он нужен для вопросов: “как защитить FastPanel”, “можно ли давать доступ к панели”, “что закрыть на сервере”, “как не потерять доступ после установки”.

Текст написан простым языком. Он не заменяет работу системного администратора, но помогает владельцу понять, что обязательно проверить.

## Главное правило

Панель сервера — это доступ к сердцу проекта. Если злоумышленник получит доступ к FastPanel, он может повредить сайт, базу, файлы, почту, резервные копии и настройки домена.

Поэтому FastPanel нельзя оставлять “как поставили по умолчанию”. Ее нужно защищать так же серьезно, как админку обменника.

## Когда использовать

| Ситуация | Что объяснить |
| --- | --- |
| Клиент только установил FastPanel | Что проверить сразу после установки |
| Клиент спрашивает, можно ли дать доступ поддержке | Как дать временный и ограниченный доступ |
| Клиент боится взлома | Какие базовые меры защиты включить |
| После обновления или установки что-то не работает | Что проверить без раскрытия паролей |
| Нужно подготовить сервер к запуску | Какие настройки должны быть закрыты до приема клиентов |

## Основные меры защиты

| Мера | Что означает простыми словами | Рекомендация |
| --- | --- | --- |
| Сильный пароль | Пароль нельзя угадать или подобрать | Использовать длинный уникальный пароль |
| Отдельный пользователь | Не работать постоянно под главным доступом | Создавать отдельные доступы под задачи |
| Ограничение по IP | Вход в панель разрешен только с доверенных адресов | Включать, если у команды стабильные IP |
| Защита входа | Дополнительная проверка при входе | Включить 2FA, если доступна |
| Закрытые порты | Снаружи доступны только нужные службы | Все лишнее закрыть |
| Обновления панели | Исправляют известные проблемы безопасности | Обновлять по регламенту и после резервной копии |
| Резервные копии | Можно восстановить сайт после ошибки | Хранить копии не только на этом же сервере |
| Журналы входов | Видно, кто и когда входил | Проверять при подозрительной активности |

## Что проверить сразу после установки FastPanel

| Проверка | Как должно быть | Почему важно |
| --- | --- | --- |
| Пароль панели | Уникальный и длинный | Нельзя использовать пароль из других сервисов |
| Доступ к панели | Не открыт для всех без ограничений | Чем меньше доступных входов, тем ниже риск |
| SSL для панели | Панель открывается по защищенному соединению | Пароль не должен передаваться небезопасно |
| Пользователи панели | Есть только нужные пользователи | Старые и временные доступы нужно удалить |
| Сайт обменника | Работает отдельно от лишних сайтов | Не храните на сервере случайные проекты |
| База данных | Не открыта публично без необходимости | Публичная база — высокий риск |
| Очереди и фоновые процессы | Запускаются и контролируются | Иначе курсы, письма и выплаты могут зависать |
| Резервные копии | Создаются и проверяются | Копия бесполезна, если ее нельзя восстановить |

## Как безопасно дать доступ специалисту

| Шаг | Что сделать |
| --- | --- |
| 1 | Создать временный доступ, если панель это позволяет |
| 2 | Дать только те права, которые нужны для задачи |
| 3 | Передать доступ через защищенный канал |
| 4 | Не отправлять пароль в общий чат |
| 5 | После завершения работы сменить пароль или удалить временного пользователя |
| 6 | Проверить журнал входов и выполненные действия |

Если специалист просит главный доступ, владелец должен понимать зачем. Иногда это действительно нужно для установки, но после работы доступ нужно пересмотреть.

## Что нельзя делать

| Нельзя | Почему |
| --- | --- |
| Использовать один пароль для панели, почты и админки | Взлом одного сервиса откроет все |
| Хранить пароли в открытом чате | Чат могут увидеть посторонние |
| Давать постоянный полный доступ всем подряд | Нельзя понять, кто что изменил |
| Оставлять тестовые аккаунты | Их могут использовать позже |
| Открывать базу данных наружу без необходимости | Это риск утечки и взлома |
| Хранить единственный backup на том же сервере | При поломке сервера копия тоже может пропасть |
| Ставить на сервер обменника случайные сайты | Чужой сайт может стать слабым местом |
| Отключать защиту ради удобства | Удобство не должно ломать безопасность |

## Защита FastPanel перед публичным запуском

1. Сменить все временные пароли.
2. Удалить лишних пользователей панели.
3. Ограничить вход по IP, если это возможно.
4. Проверить SSL для сайта и панели.
5. Закрыть лишние порты.
6. Проверить, что база данных не открыта публично.
7. Настроить резервные копии.
8. Проверить восстановление из резервной копии.
9. Проверить фоновые процессы.
10. Проверить журналы входов.

## Типовые ошибки

| Ошибка | Причина | Что сделать |
| --- | --- | --- |
| Не могу войти в FastPanel | Неверный пароль, ограничение IP, панель недоступна | Проверить доступы и обратиться к специалисту сервера |
| После ограничения IP никто не входит | Указали неправильный IP | Нужен доступ через серверного специалиста |
| Резервные копии есть, но восстановление не проверяли | Копия может быть неполной | Проверить восстановление на безопасном сценарии |
| После обновления панели сломался сайт | Обновление затронуло настройки сервера | Не делать повторные действия, собрать ошибку и передать специалисту |
| Панель открыта всем | Нет ограничения доступа | Включить ограничения и сильный пароль |

| Вопрос клиента | Ответ |
| --- | --- |
| Как защитить FastPanel? | “Смените пароль, удалите лишних пользователей, включите ограничение по IP, проверьте SSL, закройте лишние порты, настройте резервные копии и проверьте журналы входов.” |
| Можно дать доступ поддержке? | “Да, но лучше временный и только на нужную задачу. После работы удалите доступ или смените пароль.” |
| Нужно ли ограничивать IP? | “Да, если у владельца и команды стабильные IP. Но перед включением нужно убедиться, что не потеряете доступ.” |
| Можно хранить backup на сервере? | “Можно как временную копию, но надежнее хранить еще одну копию вне этого сервера.” |
| Можно ли ставить другие сайты рядом с обменником? | “Лучше не надо. Чем меньше лишних сайтов и сервисов на сервере, тем ниже риск взлома.” |
