# Запуск обменника по шагам

## Для чего нужен раздел

Этот раздел описывает полный путь запуска обменника: от подготовки сайта до тестовой заявки. Он нужен для клиентов, которые спрашивают “что делать дальше”, “как запустить обменник”, “что настроить сначала” и “как проверить, что всё работает”.

Цель — дать понятный порядок действий без программиста.

## Общая схема запуска

1. Подготовить сервер, домен и SSL.
2. Проверить вход в админку.
3. Настроить общие параметры сайта.
4. Настроить валюты.
5. Настроить направления обмена.
6. Настроить курсы.
7. Настроить резервы.
8. Настроить реквизиты, мерчанты и автовыплаты.
9. Настроить KYC, AML и безопасность.
10. Настроить уведомления.
11. Проверить публичный сайт.
12. Создать тестовую заявку.
13. Проверить журналы и ошибки.
14. Только после этого запускать клиентов.

## Этап 1. Подготовить сервер, домен и SSL

| Что сделать | Для чего | Как проверить |
| --- | --- | --- |
| Подготовить сервер | Чтобы обменник было где разместить | Сервер доступен и соответствует требованиям |
| Подготовить домен | Чтобы клиенты открывали сайт по нормальному адресу | Домен направлен на сервер |
| Настроить SSL | Чтобы сайт работал по защищенному соединению | В браузере нет предупреждения безопасности |
| Настроить резервные копии | Чтобы можно было восстановиться после ошибки | Понятно, где создается и хранится backup |
| Проверить фоновые процессы | Чтобы работали курсы, уведомления и очереди | В админке видны последние успешные запуски |

Важно: без домена можно начать подготовку, но полноценный публичный запуск лучше делать только после домена и SSL.

## Этап 2. Проверить вход в админку

| Что проверить | Что означает | Важно знать |
| --- | --- | --- |
| Логин администратора | Владелец может войти в админку | Доступ нельзя передавать всем сотрудникам |
| 2FA | Дополнительная защита входа | Для владельца и администраторов лучше включить |
| Резервный администратор | Есть второй доверенный доступ | Помогает при потере основного доступа |
| Роли | Сотрудники видят только свои разделы | Оператору не нужен полный доступ |
| Журнал входов | Можно проверить, кто входил | Полезно при подозрительной активности |

## Этап 3. Настроить общие параметры сайта

| Настройка | Что заполнить | Проверка |
| --- | --- | --- |
| Название обменника | Название сайта и бренда | Отображается на сайте и в уведомлениях |
| Контакты | Почта, Telegram, мессенджеры, график | Клиент понимает, куда обращаться |
| Правила обмена | Условия обмена, возврата, AML/KYC | Текст подтвержден владельцем |
| Языки | Нужные языковые версии сайта | Тексты не пустые |
| Кеш | Обновление публичных данных | После изменения сайт показывает новые данные |

## Этап 4. Настроить валюты

| Шаг | Что сделать | Важно знать |
| --- | --- | --- |
| 1 | Создать или проверить валюту | Валюта сама по себе не создает обмен |
| 2 | Указать код и сеть | USDT TRC20 и USDT ERC20 лучше различать явно |
| 3 | Добавить название и логотип | Клиент должен понимать, что выбирает |
| 4 | Настроить поля реквизитов | Например, кошелек, карта, банк, комментарий |
| 5 | Добавить подсказки к полям | Подсказка снижает ошибки клиентов |
| 6 | Настроить проверки формата | Маска и валидатор должны соответствовать реквизиту |
| 7 | Настроить резерв | Без резерва направление может скрываться |
| 8 | Настроить KYC и AML, если нужно | Проверки зависят от риска валюты |

## Этап 5. Настроить направления

| Шаг | Что сделать | Проверка |
| --- | --- | --- |
| 1 | Выбрать валюту “Отдаю” | Клиент видит правильную валюту |
| 2 | Выбрать валюту “Получаю” | Пара отображается в нужном порядке |
| 3 | Включить статус направления | Направление активно только после проверки |
| 4 | Настроить курс | Курс считается правильно |
| 5 | Настроить min/max | Клиент не сможет создать заявку на неподходящую сумму |
| 6 | Настроить резерв | Сумма “Получаю” покрывается резервом |
| 7 | Настроить прибыль, доп. комиссии и комиссии ПС | Итоговая сумма понятна |
| 8 | Проверить "От суммы обмена" | Крупные и малые суммы считаются ожидаемо |
| 9 | Настроить пересчет, если нужен FIX/FLOATING | Клиент понимает, когда сумма может измениться |
| 10 | Настроить ограничения и проверки | Страны, IP, KYC, лимит новичка и лимиты заявок работают по регламенту |
| 11 | Настроить инструкцию | Клиент понимает, что делать после создания заявки |
| 12 | Настроить реквизиты или мерчант | Клиент получает способ оплаты |
| 13 | Проверить публичную форму | Направление видно и заявка создается |

## Этап 6. Настроить курсы

| Тип курса | Когда использовать | Что проверить |
| --- | --- | --- |
| Ручной курс | Редкая пара или временный запуск | Значение и точность |
| Источник курса | Курс приходит из внешнего источника | Пара, дата обновления, доступность |
| Формула | Нужно добавить маржу, fallback, кросс-курс или коэффициент | Preview, теги, коэффициенты |
| Файл курсов | Курс приходит из файла партнера или таблицы | Формат, дата, группа |
| BestChange | Нужно ориентироваться на рынок обменников | Пара, top N, фильтры, исключения |
| Конкуренты | Нужно учитывать курс выбранных обменников | Ссылки, частота обновления, ограничения |

## Этап 7. Настроить резервы

| Что сделать | Почему важно |
| --- | --- |
| Указать резерв валюты | Клиент видит доступную сумму |
| Проверить резерв направления | Некоторые пары могут иметь отдельный резерв |
| Настроить файл резервов, если используется | Резерв должен обновляться автоматически |
| Проверить скрытие направления при недостатке резерва | Сайт не должен принимать невыполнимую заявку |
| Сравнить резерв с реальными средствами | Завышенный резерв опасен |

## Этап 8. Настроить реквизиты, мерчанты и автовыплаты

| Блок | Что настроить | Как проверить |
| --- | --- | --- |
| Ручные реквизиты | Валюта, лимиты, статус, поля | Клиент получает правильный реквизит |
| Мерчант приема | Провайдер, валюта, направление, лимиты | Тестовая заявка получает оплату |
| Правила выбора мерчанта | Когда какой мерчант должен выдаваться | Журнал выбора объясняет результат |
| Автовыплата | Провайдер, лимиты, резерв, статусы | Тестовая выплата проходит безопасно |
| Журналы платежей | Ошибки и ответы провайдера | Оператор видит причину сбоя |

Важно: автовыплаты нельзя включать на рабочий поток без теста на безопасной сумме.

## Этап 9. Настроить KYC, AML и безопасность

| Проверка | Что настроить | Важно знать |
| --- | --- | --- |
| KYC | Когда клиент должен пройти проверку личности | Документы и причины отказа подтверждает владелец |
| Проверка карт | Когда нужна проверка карты | Инструкция должна быть понятна клиенту |
| AML | Какие криптооперации проверяются | Порог риска и действия подтверждает владелец |
| AML-блокировка | Что происходит при высоком риске | Нужен регламент ручной проверки |
| CAPTCHA | Защита форм от спама | Ключи нельзя отправлять в чат |
| 2FA | Защита админов | Не отключать без проверки личности |
| IP-ограничения | Защита входа в админку | Ошибка может закрыть доступ сотрудникам |

## Этап 10. Настроить уведомления

| Канал | Что настроить | Как проверить |
| --- | --- | --- |
| Email клиенту | Регистрация, заявка, смена статуса, KYC, выплата | Отправить тест и проверить спам |
| Email оператору | Новая заявка, ошибка оплаты, спорный статус | Проверить событие и шаблон |
| Telegram клиенту | Если используется Telegram-сценарий | Проверить бот и тестовое сообщение |
| Telegram оператору | Уведомления о заявках и ошибках | Проверить канал, права бота, шаблон |
| Уведомления в админке | Системные события для сотрудников | Проверить, что оператор видит важные события |

## Этап 11. Проверить публичный сайт

| Что проверить | Как должно быть |
| --- | --- |
| Главная страница | Загружается без ошибки |
| Мобильная версия | Форма обмена удобна на телефоне |
| Выбор валют | Видны нужные валюты и сети |
| Направления | Видны только рабочие пары |
| Калькулятор | Считает сумму правильно |
| Ошибки формы | Понятно объясняют, что исправить |
| Правила и чекбоксы | Клиент видит условия до создания заявки |
| Контакты | Актуальны и работают |

## Этап 12. Создать тестовую заявку

1. Открыть сайт как клиент.
2. Выбрать пару, например BTC - USDT.
3. Ввести сумму в пределах лимитов.
4. Заполнить реквизиты.
5. Проверить, что заявка создается.
6. Проверить, что клиент видит инструкцию.
7. Проверить, что оператор видит заявку в админке.
8. Проверить, что пришло уведомление.
9. Проверить статус заявки.
10. Проверить журнал заявки.
11. Если есть мерчант, проверить платежный журнал.
12. Если есть автовыплата, тестировать только по безопасному регламенту.

## Этап 13. Проверить ошибки перед запуском

| Вопрос | Что должно быть |
| --- | --- |
| Валюты видны? | Да, нужные валюты отображаются |
| Направления видны? | Да, рабочие пары доступны |
| Курсы актуальны? | Да, дата обновления свежая |
| Резерв достаточный? | Да, минимум заявки покрыт |
| Реквизиты выдаются? | Да, клиент получает правильную инструкцию |
| Заявки создаются? | Да, оператор видит их в админке |
| Уведомления приходят? | Да, email и Telegram проверены |
| KYC/AML работают? | Да, если включены |
| CRON и очереди работают? | Да, последние запуски успешны |
| Журналы понятны? | Да, есть история действий |

## Что нельзя делать перед запуском

| Нельзя | Почему |
| --- | --- |
| Запускать рекламу без тестовой заявки | Реальные клиенты увидят ошибки |
| Включать все направления сразу | Сложно контролировать курсы, резервы и реквизиты |
| Завышать резерв | Можно принять заявку, которую нельзя выполнить |
| Включать автовыплаты без теста | Есть риск финансовой ошибки |
| Отключать AML/KYC ради быстрого запуска | Это снижает контроль риска |
| Давать оператору полный доступ | Можно случайно изменить критичные настройки |
| Игнорировать мобильную версию | Большая часть клиентов может заходить с телефона |

## Минимальный запуск

Если клиент хочет запуститься аккуратно, лучше начать с малого:

1. 2-5 основных валют.
2. Несколько проверенных направлений.
3. Ручной или надежный источник курса.
4. Небольшие лимиты.
5. Реальный резерв.
6. Ручная обработка выплат на первом этапе.
7. Email и Telegram для операторов.
8. Тестовые заявки.
9. Постепенное расширение направлений.

## Частые вопросы по запуску

| Вопрос | Ответ |
| --- | --- |
| Можно ли запустить без домена? | Подготовку можно начать, но публичный запуск лучше делать после домена и SSL |
| Можно ли сразу включить много направлений? | Можно, но безопаснее запускать постепенно и проверять каждую группу |
| Можно ли сначала без автовыплат? | Да, это часто безопаснее на старте |
| Нужно ли настраивать AML сразу? | Если есть криптовалютные направления и регламент требует проверки, да |
| Что проверять первым после запуска? | Новые заявки, курсы, резервы, уведомления, платежи и журналы |
