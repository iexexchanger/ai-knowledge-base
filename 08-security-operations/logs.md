# Логи и журналы

## Для чего нужен раздел

Раздел нужен для просмотра истории действий, ошибок и событий: входы пользователей, смена статусов заявок, AML-проверки, реквизиты, мерчанты, автовыплаты, KYC, реферальные события, API-запросы, сессии и системные действия.

Если вопрос касается рабочего стола, глобального поиска, аналитики или выбора нужного журнала под конкретную проблему, используйте страницу [Рабочий стол, поиск, аналитика и журнал событий](../01-administration/workspace-search-analytics-events-reference.md).

## Когда использовать

Открывайте логи, когда нужно понять, кто изменил заявку, почему не прошла оплата, почему не отправилась выплата, что ответил мерчант, почему сработал AML/KYC, кто входил в аккаунт или почему клиент получил ошибку.

## Основные возможности

| Возможность | Что означает простыми словами | Когда использовать |
|---|---|---|
| Логи входов | История авторизаций | При подозрительных входах или жалобах на доступ |
| Мульти-авторизация | История переключения между админ-аккаунтами | Когда используется несколько админских аккаунтов |
| Активные сессии | Текущие устройства пользователя | При подозрении на чужой доступ |
| Логи статусов заявок | Кто и когда менял статус | При споре по заявке |
| AML-логи | История AML-проверок | Когда заявка заблокирована или риск высокий |
| Логи реквизитов | Какие реквизиты были выданы или изменены | При споре по оплате |
| Логи мерчантов | События приёма оплаты | Если не выдались реквизиты или не пришло уведомление от провайдера |
| Логи автовыплат | Запросы к выплатам | Если выплата не ушла или вернулась ошибка |
| История оплаты | Цепочка событий мерчанта по заявке | Для глубокой диагностики оплаты |
| KYC-заявители | Записи по проверкам личности | При проблемах с верификацией |
| KYC-логи | Ответы и статусы KYC-сервисов | Если проверка зависла или отклонена |
| Реферальные логи | История начислений и действий партнёров | При вопросах по бонусам |
| Журналы платежных шлюзов | Ответы платежных шлюзов | Для диагностики интеграций |
| Очистка логов | Автоматическое удаление старых записей | Чтобы база не разрасталась бесконечно |

## Поля и настройки

| Поле / настройка | Что означает | Как правильно заполнять | Важно знать |
|---|---|---|---|
| Дата события | Когда произошло действие | Выбирайте период вокруг проблемы | Учитывайте часовой пояс сайта |
| ID заявки | Номер заявки | Указывайте точный ID или публичный ID | По ID проще всего найти цепочку |
| Пользователь | Клиент или администратор | Ищите по ID, email или имени | Не публикуйте персональные данные |
| IP-адрес | Откуда было действие | Используйте для проверки подозрительных действий | Если прокси настроен неверно, IP может быть неправильным |
| Статус до | Как было раньше | Смотрите для понимания исходного состояния | Важно при спорах по заявке |
| Статус после | Как стало после действия | Сравнивайте со статусом до | Показывает, кто перевёл заявку дальше |
| Сообщение ошибки | Текст ошибки сервиса | Читайте полностью | Не копируйте наружу секретные детали |
| Данные обмена с провайдером | Что отправлено провайдеру и какой ответ получен | Используйте для диагностики мерчанта | Может содержать чувствительные данные |
| Уровень события | Информация, предупреждение, ошибка | Ошибки важнее для диагностики | Информационные события не всегда проблема |
| Источник | Откуда пришло событие | Мерчант, автовыплата, CRON, оператор, клиент | Помогает найти виновный участок цепочки |
| Включить очистку логов | Удалять старые записи автоматически | Включайте после согласования срока хранения | Не удаляйте логи, нужные для финансовых споров |
| Срок хранения | Сколько дней или месяцев хранить | Выставляйте по регламенту обменника | Для платежей и KYC может требоваться больше |
| Защитный буфер | Дополнительные дни перед удалением | Указывайте неотрицательное число | Защищает свежие записи от случайной очистки |

## Как настроить

1. Откройте нужный журнал.
2. Найдите событие по ID заявки, пользователю, дате или источнику.
3. Сравните цепочку: заявка, статус, реквизиты, мерчант, AML/KYC, выплата.
4. Если проблема связана с платежом, проверьте логи мерчанта и история оплаты.
5. Если проблема связана с выплатой, проверьте логи автовыплат и статус заявки.
6. Если проблема связана с клиентом, проверьте входы, IP, сессии и блокировки.
7. В настройках очистки логов задайте срок хранения для каждого типа журнала.

## Типовые ошибки

| Ошибка / ситуация | Причина | Что сделать |
|---|---|---|
| Логов нет | Событие старое, очистка уже удалила записи или журнал выключен | Проверить настройки очистки и период поиска |
| В логах один IP у всех клиентов | Неверно настроены доверенные прокси | Исправить прокси в безопасности |
| Мерчант вернул ошибку | Неверные настройки, лимиты, сеть, реквизиты или недоступность сервиса | Проверить мерчант, валюту, направление и журналы платежных шлюзов |
| AML-проверка не понятна | Не выбран сервис или нет ответа провайдера | Проверить AML-сервис и логи |
| Статус заявки поменялся неожиданно | Автоматизация, оператор или CRON | Проверить статус-логи и процессы |
| Логи занимают много места | Долгий срок хранения | Настроить очистку с безопасным сроком |

## Важные предупреждения

Логи могут содержать чувствительные данные: реквизиты, email, IP, ответы мерчантов, статусы проверок, фрагменты запросов. Не добавляйте такие данные в публичные тикеты, документацию или сообщения клиентам. Для клиента объясняйте простыми словами: “платёж не подтвердился”, “провайдер вернул ошибку”, “требуется ручная проверка”.

Не очищайте логи без согласования, если есть спорные заявки, бухгалтерская проверка, проблема с AML/KYC или расследование по доступу.
